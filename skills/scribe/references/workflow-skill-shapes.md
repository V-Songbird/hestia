# Workflow-skill target shapes

When decomposition (see [decomposition.md](decomposition.md)) routes a piece of the source artifact to a skill, this file defines the target shape. Decomposition produces *what* and *where*; this file produces *how the artifact looks*.

## 1. Reference vs action skills

The [features overview](https://code.claude.com/docs/en/features-overview#skill-vs-subagent) splits skills into two categories. The decomposition phase routes by content shape; the resulting skill must match its category in body shape too.

| Category | Body shape | Frontmatter signature | Example |
|---|---|---|---|
| **Reference skill** | Standing instructions, conventions, knowledge Claude applies throughout the session. No imperative top-level steps. | `description` describes the *knowledge*; usually no `disable-model-invocation`; never `context: fork`. | API style guide; role catalogs; integration-contract template |
| **Action skill** | Imperative top-level steps (`Find â€¦`, `Dispatch â€¦`, `Generate â€¦`). Has a clear start and end. | `description` describes the *action*; usually `disable-model-invocation: true` for sequenced workflows; can use `context: fork` if no main-session context is needed. | `/expert-analysis`, `/critic-review`, `/deploy` |

**Mechanical rule:** an action skill's body has top-level steps with imperative verbs. A reference skill's body has section headings that name *knowledge categories* (`## Conventions`, `## Patterns`, `## Schemas`). Mismatched body + frontmatter is a structural fail.

## 2. The 7-section layout for action skills

Workflow-orchestrator action skills (the kind decomposition produces from a multi-step source) follow this exact layout. Order matters â€” this is what Claude scans on invocation.

```
1. Frontmatter                  (5â€“8 lines)
2. # <Skill name>               (1 line)
3. One-sentence purpose         (1â€“2 lines)
4. ## Required Inputs           (2â€“5 lines, bulleted)
5. ## Dispatch Template         (15â€“30 lines, fenced Agent(...) block)
6. ## Critical Constraints      (3â€“5 bullets)
7. ## Next Step                 (1â€“2 lines, names the next /slash-command)
8. ## Additional resources      (optional, 1â€“3 bullets pointing to references/)
```

**Why this order:**
- Purpose first establishes the contract.
- Inputs second tells the caller what's needed before invoking.
- Dispatch template is the operational core â€” what Claude copies and customizes.
- Constraints prevent foot-guns the template alone cannot encode.
- Handoff creates workflow continuity (Claude knows what comes next without re-reading CLAUDE.md).
- References enable detail-on-demand for stack-specifics or templates.

**Target body length:** < 100 lines. If exceeded, extract templates to `references/` per [decomposition.md Â§ 3](decomposition.md).

## 3. When to set each frontmatter field

For action skills generated by decomposition. (Reference skills usually have only `name` + `description`.)

### `disable-model-invocation`

| Set to `true` when â€¦ | Set to `false` (or omit) when â€¦ |
|---|---|
| Skill is a sequenced workflow step (auto-firing breaks the sequence) | Skill is reference-shaped or always-safe-to-trigger |
| Skill has side effects (commits, deploys, dispatches subagents) | Skill is read-only knowledge or analysis |
| Skill is destructive or expensive (per-invocation cost matters) | Cost is negligible |

For decomposed workflow skills: **always `true`**. Auto-firing `/critic-review` before experts have reported breaks the workflow; auto-firing `/developer-dispatch` is actively dangerous.

### `allowed-tools`

Pre-approves tools while the skill is active. Does **not** restrict what's callable â€” that's `permissions.deny` in `settings.json`.

Compose by union:
- Always: `Read Grep Glob` (every workflow skill reads code)
- If body dispatches subagents (`Agent(...)` block present): add `Agent`
- If body uses dynamic injection (`` !`<cmd>` ``): add `Bash(<scoped> *)` â€” narrowest pattern that covers the injections (e.g. `Bash(git *)`, not `Bash(*)`)
- If body produces files: add `Write Edit`

### `model`

The override applies for the rest of the current turn only; the session model resumes on the next prompt. `inherit` keeps the active model.

**For decomposed workflow skills:** set on every skill where the source artifact specified a model. The source workflow's model assignment table is policy â€” skills must encode it, not leave it dependent on user session state. Without explicit `model:`, a user on Sonnet invoking `/critic-review` gets Sonnet doing adversarial review, contradicting the workflow's design intent.

Mapping from source `**Model**: <X>` annotation:
- `Opus` â†’ `model: opus`
- `Sonnet` â†’ `model: sonnet`
- `Haiku` â†’ `model: haiku`

**Synthesis â‡’ opus standing rule (no source annotation required).** When the step is synthesis-shaped â€” matched by the same regex as `effort: high` (`synthesize`, `consolidate`, `master plan`, `ground-truth`, `exhaustively defy`) â€” default `model: opus` even if the source omits the annotation. Synthesis quality is model-bound, not effort-bound: `effort: high` on Sonnet still produces a Sonnet-shaped synthesis. This rule applies symmetrically to subagent definitions whose body is a synthesis prompt (`adversarial-critic`, `master-plan-synthesizer`, `expert-revise`).

The same standing rule applies in reverse: parallel-worker / role-injected dispatcher skills (e.g. `expert-analysis`, `developer-dispatch`) default to `model: sonnet` unless the source specifies otherwise. Opus on a parallel dispatcher is wasted cost â€” quality lives in the workers' synthesis pass, not in the mechanical dispatch.

### `effort`

Set `high` (or `xhigh` if model supports it) on synthesis / consolidation skills where reasoning depth most affects output quality. Detection: step prose contains "synthesize", "consolidate", "ground-truth", "master plan", "exhaustively defy".

For dispatcher skills (parallel role-injected workers, mechanical work-unit mapping), omit `effort` â€” added cost without quality gain.

### `maxTurns` (subagent definitions and `Agent` dispatches)

`maxTurns` MUST be derived, not picked by feel. Stating the derivation in a comment beside the field lets reviewers question it.

**Default formula:**

```
maxTurns = (numbered workflow steps in body) Ã— 2 + 4 safety margin, rounded UP to nearest 5, capped at 30
```

The `Ã— 2` covers one input turn + one tool/output turn per step. The `+4` absorbs retries, recovery, and the return-format pass.

| Body shape | Steps | Computed | Rounded |
|---|---|---|---|
| Tight read-only researcher (3-step list-and-summarize) | 3 | 10 | **10** |
| Typical 5â€“6 step audit / dispatcher | 5â€“6 | 14â€“16 | **15** |
| Multi-phase synthesis with iteration loop | 8â€“10 | 20â€“24 | **25** |
| Heavy refactor worker | 12+ | 28+ | **30** (capped) |

**Cap rule:** any subagent needing > 30 turns is doing too much â€” split it, or move iteration into the dispatching session.

**Adjustments:**
- **Synthesis-shaped subagent:** add +5 to the computed value (synthesis often re-reads its own draft).
- **Background subagent (`background: true`):** subtract âˆ’5 (background workers should fail-fast rather than loop).
- **Bounded-input researcher (`Read`/`Grep`/`Glob` only, no iteration):** clamp to 10 regardless of formula.

When the source artifact does not list discrete numbered steps (e.g. an inline prompt body), count distinct *actions* the subagent performs end-to-end (read X, search Y, decide, emit) and apply the same formula.

### `argument-hint` and `arguments`

`argument-hint` shows in autocomplete (e.g. `[concern-number]`). `arguments` declares named positional arguments for `$name` substitution in the body.

Set when the skill takes per-invocation parameters: concern number, work-unit number, fix index. Don't set for skills whose entire input is the prior conversation context.

### `context: fork`

**Never set on workflow-orchestrator skills.** Two disqualifications:

1. *"It won't have access to your conversation history"* â€” synthesis-shaped skills (`master-plan`, `expert-revise`) need expert/critic reports already in conversation; forking would lose them.
2. *"Subagents cannot spawn other subagents"* â€” every dispatcher skill needs to spawn workers via `Task`. Forking would prevent that.

`context: fork` is for **standalone task skills** that have everything they need in their own body and produce a self-contained result. Not for orchestration.

### `paths`

Set when the skill is path-scoped (e.g. an audit skill that only applies to `src/api/**`). Auto-activation honors this; manual `/<name>` invocation ignores it.

For workflow skills: usually omit. Workflow skills apply to the whole project, not specific paths.

## 4. CLAUDE.md hub specification

Decomposition produces a thin CLAUDE.md whose sole job is to teach Claude **when** to enter the workflow and **which** slash command to invoke at each step. The body is fixed-shape:

```markdown
# <Workflow name>

## Workflow Triggers
<one-paragraph rule for "when to enter the workflow", with example phrases>

## The N Steps
| # | Action | Model | Skill |
|---|--------|-------|-------|
| 1 | <Action description> | <Opus/Sonnet/Haiku, or em-dash if direct> | <`/slash-command`, or em-dash if no skill> |
| ... |

## <Approval / risk / gating logic>
<thresholds, e.g. Low / Medium / High risk criteria>

## Build Commands
<only commands the user runs directly; not internal step commands>

## Workflow Principles
- <Standing rule 1, e.g. "Citations required: experts and critic cite file:line, not summaries">
- <Standing rule 2, e.g. "Contract before code: integration contract precedes any developer dispatch">
- <Standing rule 3, e.g. "Escalate blockers: developers report contradictions; orchestrator decides">
- <Standing rule 4, e.g. "Parallel + isolated: isolation: \"worktree\" on developer dispatch">

## Detailed Reference
For step-by-step detail, edge cases, and example flows, see `<workflow-reference>.md`.
```

**Mechanical exclusions** (these get extracted elsewhere, not embedded in CLAUDE.md):
- Full `Agent(...)` prompt bodies â†’ live in skill SKILL.md files
- Templates (integration contract, plan structure, blocker decision tree) â†’ live in skill `references/`
- Stack-specific customization â†’ lives in `references/<name>-by-stack.md`

**Soft cap:** â‰¤ 200 lines per [memory docs](https://code.claude.com/docs/en/memory). Decomposed CLAUDE.md should be 30â€“80 lines.

## 5. The Skill+Subagent combination pattern

The [features overview combination patterns](https://code.claude.com/docs/en/features-overview#combine-features) name this explicitly: *"`/audit` skill kicks off security, performance, and style subagents that work in isolated context."*

This is the dominant pattern for decomposed workflow skills:

```
Skill body (action skill)
â””â”€â”€ ## Dispatch Template
    â””â”€â”€ Agent(
          subagent_type: "<custom-or-general>",
          ...
        )
        â†“ dispatches to
        Subagent body (fixed role, locked tools)
        â””â”€â”€ system prompt + tools + maxTurns
            â†“ returns
            Structured report â†’ main session
```

**Recognition rule for proofreader:** a skill whose body contains a `## Dispatch Template` with `Agent(...)` is the documented combination pattern. Do not flag it as "skill doing too much" â€” orchestration is the intended shape.

**When to extract the inner role into a custom subagent (not just `subagent_type: "general-purpose"` with role in prompt):**
- The role is fixed across invocations (`adversarial code reviewer`, `documentation auditor`)
- The role needs tool restrictions (`Read Grep Glob` only, no `Write`)
- The role needs bounded `maxTurns`
- The role is reused across multiple skills

hestia's own `proofreader` is the canonical example: fixed role, locked tools (`Read Grep Glob`), `maxTurns: 12`, dispatched from `scribe`.

## 6. The "what NOT to include" checklist

When generating workflow skills, deliberately omit:

| Omit | Reason |
|---|---|
| Lengthy preambles ("This powerful skill helps you â€¦") | Marketing language taxes context for zero signal |
| Full workflow re-explanation in every SKILL.md | That's CLAUDE.md's job; per-skill repetition is waste |
| Multiple variants of the same template | Pick one, reference others in `references/` |
| Inline templates > 15 lines (contracts, plans, output formats) | Extract to `references/` and link |
| Stack-specific role examples in the dispatch template | Extract to `references/<name>-by-stack.md` |
| `model:` field omitted "because the user can pick" | Workflow design specifies the model; encode it |
| `disable-model-invocation` omitted on a sequenced workflow step | Auto-firing breaks the sequence |
| Persistent state files (caches, decision logs) absent from the source artifact | Don't invent what wasn't asked for; state risks staleness |
